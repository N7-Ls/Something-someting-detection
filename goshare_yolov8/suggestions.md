## 系統架構與併發邏輯優化建議

本專案採用多執行緒 (Threading) 與 Queue 分離擷取、推論、決策與 UI 的架構，有效避免了單一耗時任務卡死視訊串流。針對目前的程式結構，有以下五點優化建議：

### 1. 影格同步與畫面撕裂 (State Synchronization)
* **現況**：全域字典 `display_state` 由多個執行緒非同步寫入。由於 YOLO 處理速度慢於 MediaPipe，UI 渲染時可能會將不同幀的 BBox 與網格畫在同一畫面上，導致快速移動時標註無法對齊。
* **建議**：若需嚴格的視覺對齊，應將狀態儲存改為「帶有 Frame ID 的快照」，UI 端統一依據最新的 Frame ID 組合對應的狀態進行渲染。

### 2. 丟幀策略與推論頻率 (Drop Frame Strategy)
* **現況**：Queue 滿時的丟棄機制是維持 Real-time 的好作法，但會導致 YOLO 執行緒內的 `cig_frame_counter` 與實際攝影機影格數脫鉤。這會使 `CIG_SKIP_FRAMES` 的推論間隔被意外拉長（例如從 0.1 秒變成 0.6 秒）。
* **建議**：改用時間間隔 (Time Interval) 來控制推論，或直接依賴 `frame_id % CIG_SKIP_FRAMES == 0` 來判斷，確保動作捕捉的頻率穩定。

### 3. 計時器函數選擇 (Timer Functions)
* **現況**：決策中心使用 `time.time()` 計算分心或抽菸的持續時間。這屬於系統絕對時間，若硬體設備觸發 NTP 網路校時或發生微小時間跳動，會導致時間差計算出現極大正負值，進而引發系統誤判。
* **建議**：凡涉及「持續時間」或「時間差」的計算，一律改用 `time.perf_counter()`，以確保獲得高解析度且不受系統時間更改影響的單調時鐘。

### 4. GIL 限制與效能瓶頸 (GIL & CPU Bottlenecks)
* **現況**：底層 C++ 函式庫 (OpenCV, YOLO, MediaPipe) 在推論時會釋放 Python 的全局直譯器鎖 (GIL)，因此具有一定平行運算能力。但 `thread_decision` 中的陣列、字典操作與 CSV 寫入，以及 UI 的 `cv2.addWeighted` 渲染仍受 GIL 限制。
* **建議**：若未來部署於算力較低的邊緣運算設備，這兩個純 Python 執行緒可能會成為效能瓶頸，屆時可評估將架構重構為多處理程序 (Multiprocessing)。

### 5. 決策融合的極端情況 (Decision Fusion Edge Cases)
* **現況**：決策融合邏輯中，若 `cache_yolo` 無當前幀資料，會自動取用 `latest` 歷史紀錄。若 YOLO 執行緒因故卡死，決策中心會持續拿「過期的 YOLO 狀態」與「最新的 MediaPipe 狀態」融合，可能引發「幻影抽菸」等誤觸發。
* **建議**：對 `latest` 快照增加過期時間的檢驗邏輯（例如 `now - latest["timestamp"] > 0.5` 即判定為無效），避免過期資料干擾決策。

## 模型部分優化

### 程式碼暴力過濾（物理限制與動態 ROI）

在不重新訓練模型的前提下，透過整合 MediaPipe 與 YOLO 的結果，寫死物理規則來濾除荒謬的誤判（例如將騎士的外套皺褶判斷為手機）。

**1. 面積與長寬比限制 (BBox Constraints)**
* **邏輯**：手機在畫面中的佔比有物理極限，且形狀具備特定特徵。
* **實作**：在 YOLO 執行緒取得 BBox 後，立即計算其面積與長寬比。若 BBox 面積超過畫面總面積的 10%，或寬度大於騎士的肩膀寬度，則直接丟棄 (`continue`)。若寬高比接近 1:1（極可能是身體軀幹），亦視為誤判並濾除。

**2. 動態感興趣區域裁切 (Dynamic ROI Crop)**
* **邏輯**：抽菸與操作手機的行為高度集中在手部與臉部周圍，無需讓模型掃描騎士的腹部或腿部。
* **實作**：先利用 MediaPipe 取得臉部中心座標，向下及向左右延伸出一個涵蓋「頭頂至胸部」的矩形區域 (ROI)。將此 ROI 影像獨立裁切出來，再送入 YOLO 進行推論。此舉能 100% 杜絕模型掃描到身體下半部產生的誤判，同時相對放大了香菸在送測影像中的像素佔比。

---

### 資料集微調 Fine-tuning（解決領域偏移）

若開源預訓練模型（通常以室內、清晰特寫為主）無法適應共享機車儀表板視角（戶外光影、穿著外套/雨衣、戴安全帽），則必須進行少量的轉移學習 (Transfer Learning) 來適應目標場景。

**1. 收集真實場景資料 (Custom Dataset)**
* **作法**：實際架設於機車儀表板視角，錄製騎士戴安全帽、穿著不同外套，在真實環境下操作手機與抽菸的短影片，並從中擷取影像進行標註。

**2. 引入負樣本 (Negative Samples) - 關鍵步驟**
* **作法**：大量擷取「無手機、無香菸」的正常騎乘畫面（包含各種手勢、反光、大面積外套皺褶），且**不標註任何 BBox**，直接混入資料集作為背景圖 (Background Images)。
* **目的**：強迫 YOLO 模型學習並區分「這些輪廓（如安全帽反光、後照鏡、拉鍊）並非目標物」，從根本上降低把人體誤判為手機的機率。

**3. 轉移學習訓練 (Transfer Learning)**
* **作法**：以 Hugging Face 下載的輕量化模型權重 (如 `yolov8n.pt` 或特化模型) 作為 Pre-trained Weights。使用上述準備好的 100~200 張真實場景照片與負樣本，進行小規模的 Epoch 訓練（約 50 Epochs 即可），更新模型特徵層以適應實際的車機視角。
